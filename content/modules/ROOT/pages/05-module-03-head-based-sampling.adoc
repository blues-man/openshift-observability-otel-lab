= Module 3: Head-Based Sampling

****
*Duration:* ~15 minutes
****

Head-based sampling makes the decision at the *start* of a trace, before any outcomes are known. It is simple and low-overhead, but it is blind to whether a trace contains errors or high latency. In this module you will configure head-based sampling at the SDK level through the `Instrumentation` CR, explore collector-level probabilistic sampling, and observe the impact on span volume.

== Exercise 1: Understand Head-Based Sampling

Head-based sampling decides whether to record a trace *at creation time*. The decision is encoded in the https://www.w3.org/TR/trace-context/#traceparent-header[W3C `traceparent` header] and propagated to every downstream service in the request chain. No span in the trace can override the original decision.

=== Key Samplers

[cols="1,3",options="header"]
|===
| Sampler | Behaviour

| *AlwaysOn*
| Records 100% of traces. Default in most SDKs.

| *AlwaysOff*
| Records 0% of traces. Useful for disabling tracing without removing instrumentation.

| *TraceIdRatioBased / ProbabilitySampler*
| Records a configurable percentage of traces based on the trace ID hash.

| *ParentBased*
| Wraps another sampler and respects the parent span's sampling decision. Root spans use the wrapped sampler; child spans follow the parent.
|===

=== When to Use Head-Based Sampling

* You need *uniform cost reduction* across all services.
* You do not need outcome-aware decisions (errors, latency).
* You want minimal processing overhead -- the decision is a single check at trace creation.

=== When NOT to Use Head-Based Sampling

* You must keep *all error traces* regardless of percentage.
* You need to retain *slow traces* above a latency threshold.
* You require adaptive sampling that reacts to system conditions.

For those scenarios, *tail-based sampling* (covered in the next module) is the better choice.

== Exercise 2: SDK-Level Sampling

The current `Instrumentation` CR has `sampler.argument: "1"` which means 100% of traces are sampled. You will reduce this to 50%.

=== Step 1 -- Apply 50% Sampling

Apply an updated `Instrumentation` CR that sets `parentbased_traceidratio` with an argument of `0.5`:

[source,subs="attributes+",role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: my-instrumentation
  namespace: {tutorial_namespace}
spec:
  propagators:
    - tracecontext
    - baggage
    - b3
  sampler:
    type: parentbased_traceidratio
    argument: "0.5"
  python:
    env:
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: "http://{collector_name}.{tutorial_namespace}.svc:4318"
  nodejs:
    env:
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: "http://{collector_name}.{tutorial_namespace}.svc:4318"
  addK8sUIDAttributes: true
EOF
----

Expected output:

----
instrumentation.opentelemetry.io/my-instrumentation configured
----

=== Step 2 -- Restart Application Pods

The sampling configuration is injected at pod start time. Restart the application deployments to pick up the new value:

[source,subs="attributes+",role="execute"]
----
oc rollout restart deployment/frontend deployment/backend1 deployment/backend2 -n {tutorial_namespace}
----

Wait for the rollout to complete:

[source,subs="attributes+",role="execute"]
----
oc rollout status deployment/frontend -n {tutorial_namespace}
----

Expected output:

----
deployment "frontend" successfully rolled out
----

=== Step 3 -- Verify the Span Rate Drop

Open the OpenShift Console and navigate to *Observe -> Metrics*. Run the following PromQL query:

[source,role="copypaste"]
----
sum(rate(telemetry_spans_count_total[5m]))
----

You should see the span rate drop by approximately 50% compared to the baseline.

== Exercise 3: Collector-Level Probabilistic Sampling

In addition to SDK-level sampling, you can sample at the *collector* level using the `probabilistic_sampler` processor. This is useful for services you cannot modify -- third-party workloads, legacy applications, or sidecar-injected services.

=== Collector Configuration

The following snippet shows how the processor is configured inside an `OpenTelemetryCollector` CR pipeline:

[source,yaml]
----
processors:
  probabilistic_sampler:
    mode: equalizing
    sampling_percentage: 50
----

=== Sampling Modes

[cols="1,3",options="header"]
|===
| Mode | Behaviour

| *equalizing*
| Reads the upstream `th:` tracestate key. If the incoming sampling rate is already at or below the target percentage, the span passes through unchanged. Otherwise the processor reduces to the target. This is the recommended mode when SDK-level sampling is also active.

| *hash_seed*
| Ignores any upstream sampling decision and applies an FNV hash on the trace ID. Can *oversample* if upstream already sampled below the target.

| *proportional*
| Applies a strict proportional reduction relative to the incoming rate. Produces the most predictable volume but requires accurate upstream rate metadata.
|===

[IMPORTANT]
====
We will *not* apply collector-level sampling in this exercise because SDK-level sampling is already active. The two approaches *stack multiplicatively*: 50% SDK x 50% collector = 25% actual throughput. Keep this interaction in mind when designing production pipelines.
====

== Learning Outcomes

By completing this module you can:

* Explain head-based sampling trade-offs -- simple and low-overhead, but blind to trace outcomes such as errors or high latency.

* Configure SDK-level sampling via the `Instrumentation` CR using `parentbased_traceidratio`.

* Identify when collector-level probabilistic sampling is appropriate -- third-party or legacy services where SDK configuration is not possible.

* Describe how SDK and collector sampling interact multiplicatively and plan accordingly for production pipelines.
