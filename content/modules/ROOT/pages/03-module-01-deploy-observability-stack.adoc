= Module 1: Explore the Observability Stack

[cols="1h,3"]
|===
| Duration | ~20 minutes
| Objective | Verify the pre-deployed observability stack and understand how each component fits the architecture
|===

The full observability stack -- operators, backends, and console integrations -- has been pre-deployed to this cluster via ArgoCD GitOps.
In this module you will walk through every layer to confirm it is healthy and understand how the pieces connect before you start generating telemetry.

== Getting Started: Log In to OpenShift

Before running any commands in this lab, you need to log in to the OpenShift cluster.

[TIP]
====
Use the *Terminal* tab on the right side of this tutorial window to run commands. When an exercise asks you to open the OpenShift Console, switch to the *OCP Console* tab instead.
====

Run the following command in the *Terminal*:

[source,subs="attributes+",role="execute"]
----
{login_command}
----

You should see output similar to:

----
Login successful.

You have access to N projects, the list has been suppressed.

Using project "observability-app-userN".
----

Verify you are connected:

[source,subs="attributes+",role="execute"]
----
oc whoami
----

Expected output:

[source,bash,subs="attributes+"]
----
{user_name}
----

Once you see your username, you are ready to proceed with the exercises below.

== Exercise 1: Verify Operators

Five operators power the observability stack.
Confirm that each Subscription exists and that every ClusterServiceVersion has reached the *Succeeded* phase.

=== Step 1 -- Check Subscriptions

[source,role="execute"]
----
for ns in openshift-opentelemetry-operator openshift-tempo-operator openshift-operators-redhat openshift-cluster-observability-operator openshift-logging; do
  oc get subscriptions.operators.coreos.com -n $ns 2>/dev/null
done
----

You should see output similar to:

----
NAME                       PACKAGE                       CHANNEL   ...
opentelemetry-product      opentelemetry-product         stable    ...
NAME                       PACKAGE                       CHANNEL   ...
tempo-product              tempo-product                 stable    ...
NAME                       PACKAGE                       CHANNEL   ...
loki-operator              loki-operator                 stable-6.5 ...
NAME                       PACKAGE                       CHANNEL   ...
cluster-observability-operator  cluster-observability-operator  stable ...
NAME                       PACKAGE                       CHANNEL   ...
cluster-logging            cluster-logging               stable-6.5 ...
----

=== Step 2 -- Check CSVs

[source,role="execute"]
----
for ns in openshift-opentelemetry-operator openshift-tempo-operator openshift-operators-redhat openshift-cluster-observability-operator openshift-logging; do
  oc get csv -n $ns 2>/dev/null | grep -E 'opentelemetry|tempo|loki|observability|logging'
done
----

Every CSV should show *Succeeded* in the `PHASE` column:

----
opentelemetry-operator.v...   Succeeded
tempo-operator.v...           Succeeded
loki-operator.v...            Succeeded
cluster-observability...      Succeeded
cluster-logging.v...          Succeeded
----

=== What each operator provides

* *Red Hat build of OpenTelemetry Operator* -- manages OpenTelemetry Collector instances (sidecars, deployments, DaemonSets) and the Instrumentation CR for auto-instrumentation.

* *Tempo Operator* -- deploys and manages TempoStack, the distributed tracing backend that replaces standalone Jaeger.

* *Loki Operator* -- deploys and manages LokiStack, the horizontally-scalable log aggregation backend.

* *Cluster Observability Operator (COO)* -- provides MonitoringStack CRs and UIPlugin CRs for native OpenShift Console integrations (traces, logs, dashboards).

* *Red Hat OpenShift Logging Operator* -- manages LokiStack (the log aggregation backend) and the optional ClusterLogForwarder for cluster-wide container log collection. In this lab, application logs from instrumented services are sent to LokiStack primarily through the OpenTelemetry Collector using OTLP.

== Exercise 2: Explore the Tracing Backend (TempoStack)

TempoStack is the OpenShift-native distributed tracing backend.
It stores traces in S3-compatible object storage -- in this lab a MinIO instance provides that storage.

=== Step 1 -- Inspect the TempoStack CR

[source,subs="attributes+",role="execute"]
----
oc get tempostack -n {tempo_namespace}
----

Expected output:

----
NAME       AGE    VERSION   STATUS   STORAGE SECRET
simplest   ...    ...       Ready    minio-secret
----

The *Ready* status confirms that Tempo is up and connected to its storage backend.

=== Step 2 -- Check Tempo pods

[source,subs="attributes+",role="execute"]
----
oc get pods -n {tempo_namespace}
----

You should see pods for the Tempo compactor, distributor, ingester, querier, query-frontend, and the gateway:

----
NAME                                        READY   STATUS    RESTARTS   AGE
tempo-simplest-compactor-...                1/1     Running   0          ...
tempo-simplest-distributor-...              1/1     Running   0          ...
tempo-simplest-ingester-...                 1/1     Running   0          ...
tempo-simplest-querier-...                  1/1     Running   0          ...
tempo-simplest-query-frontend-...           1/1     Running   0          ...
tempo-simplest-gateway-...                  1/1     Running   0          ...
----

=== Step 3 -- Check the Tempo route

[source,subs="attributes+",role="execute"]
----
oc get routes -n {tempo_namespace}
----

----
NAME                      HOST/PORT                               ...
tempo-simplest-gateway    tempo-simplest-gateway-tempo-...        ...
----

The gateway route exposes the Jaeger Query-compatible UI and the gRPC/HTTP endpoints that OpenTelemetry Collectors send traces to.

[NOTE]
====
A *UIPlugin* (covered in Exercise 5) integrates the Tempo query interface directly into the OpenShift Console under *Observe -> Traces*, so you rarely need to access the raw route.
====

== Exercise 3: Explore the Logging Backend (LokiStack)

LokiStack provides horizontally-scalable log aggregation and serves as the central logging backend in this lab.
Two complementary paths send logs to LokiStack:

* The *OpenTelemetry Collector* pushes application logs from instrumented services directly to LokiStack using the OTLP protocol -- this is the primary path you will work with throughout the lab.
* The Logging Operator's *ClusterLogForwarder* collects container-level application and infrastructure logs from cluster nodes and forwards them to LokiStack for platform-wide coverage.

=== Step 1 -- Inspect the LokiStack CR

[source,subs="attributes+",role="execute"]
----
oc get lokistack -n {loki_namespace}
----

Expected output:

----
NAME           CONDITION TYPE   CONDITION STATUS   STATUS
logging-loki   Ready            True               ...
----

=== Step 2 -- Inspect the ClusterLogForwarder

[source,subs="attributes+",role="execute"]
----
oc get clusterlogforwarder -n {loki_namespace}
----

----
NAME        AGE
collector   ...
----

View its configuration:

[source,subs="attributes+",role="execute"]
----
oc get clusterlogforwarder collector -n {loki_namespace} -o yaml | head -40
----

Key things to notice in the output:

* *inputs* -- which log types are collected (application, infrastructure, or both).

* *outputs* -- the LokiStack endpoint where logs are forwarded.

* *pipelines* -- the rules that connect inputs to outputs.

=== How the pieces fit together

This lab uses two complementary logging paths.
The *OpenTelemetry Collector* receives application logs via the OTLP protocol from auto-instrumented services and exports them directly to LokiStack -- this is the path you will configure and optimize in later modules.
In parallel, the *ClusterLogForwarder* deploys a DaemonSet that reads container logs from every node and forwards application and infrastructure logs to the same LokiStack instance in `{loki_namespace}`, providing cluster-wide log coverage.

== Exercise 4: Explore Monitoring

OpenShift ships with a built-in Prometheus stack for platform metrics.
In this lab two complementary monitoring systems are configured:

* *User Workload Monitoring (UWM)* -- the built-in OpenShift monitoring solution based on Prometheus. When enabled, a dedicated Prometheus instance scrapes `ServiceMonitor` and `PodMonitor` targets in user namespaces using the traditional *pull* model. UWM is tightly integrated with the platform and provides the familiar *Observe -> Metrics* and *Observe -> Alerting* console views.

* *Cluster Observability Operator (COO) MonitoringStack* -- a separately managed monitoring stack that exposes an OTLP HTTP receiver. This lets the OpenTelemetry Collector *push* metrics directly using the native OTLP protocol, eliminating the need for a Prometheus remote-write adapter. COO also manages the UIPlugin CRs that add the *Traces* and *Logs* views to the console.

In short, UWM handles traditional Prometheus scrape-based collection, while the COO MonitoringStack provides an OTLP-native ingestion point purpose-built for OpenTelemetry pipelines.

=== Step 1 -- Verify User Workload Monitoring

[source,role="execute"]
----
oc get configmap cluster-monitoring-config -n openshift-monitoring -o yaml
----

Look for `enableUserWorkload: true` in the output:

[source,yaml]
----
apiVersion: v1
data:
  config.yaml: |
    enableUserWorkload: true
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
----

When User Workload Monitoring is enabled, a separate Prometheus instance scrapes ServiceMonitors and PodMonitors in user namespaces.

=== Step 2 -- Check the COO MonitoringStack

[source,subs="attributes+",role="execute"]
----
oc get monitoringstack -n {coo_namespace}
----

----
NAME                AGE
coo-observability   ...
----

Inspect its specification:

[source,subs="attributes+",role="execute"]
----
oc get monitoringstack coo-observability -n {coo_namespace} -o yaml | grep -i -A2 otlp
----

You should see an OTLP HTTP receiver enabled in the MonitoringStack spec:

----
    enableOtlpHttpReceiver: true
----

[TIP]
====
The OTLP HTTP receiver lets the OpenTelemetry Collector push metrics directly to the COO MonitoringStack using the native OpenTelemetry protocol -- no Prometheus remote-write adapter needed.
====

== Exercise 5: Verify Console Plugins

The Cluster Observability Operator registers *UIPlugins* that add observability views to the OpenShift Console.

The UIPlugins are cluster-scoped resources managed by the Cluster Observability Operator.
You can confirm they are active by opening the OpenShift Console and checking for the integrated observability views.

Switch to the *OCP Console* tab on the right side of this tutorial window, then navigate to each of the integrated observability views:

. *Observe -> Logs* -- powered by the LokiStack UIPlugin integration.
Browse application and infrastructure logs using LogQL queries.

. *Observe -> Traces* -- powered by the TempoStack UIPlugin integration.
Search for traces by service name, operation, duration, or trace ID.

image::console-observe-logs.png[OpenShift Console - Observe Logs view,link=self,window=blank]

image::console-observe-traces.png[OpenShift Console - Observe Traces view,link=self,window=blank]


If these views appear under *Observe*, the `logging` and `distributed-tracing` UIPlugins are working correctly.

[NOTE]
====
If the console plugins are not yet visible, refresh the browser.
The plugins register asynchronously and may take a moment after operator deployment.
====

== Learning Outcomes

After completing this module you should be able to:

* Identify the five operators that compose the OpenShift observability stack and explain each operator's role.

* Describe how *TempoStack* stores and queries distributed traces using S3-compatible object storage.

* Explain how application logs reach *LokiStack* through the *OpenTelemetry Collector* (OTLP) and how the *ClusterLogForwarder* supplements this with cluster-wide container log collection.

* Understand how the COO *MonitoringStack* accepts metrics via its OTLP HTTP receiver.

* Navigate the OpenShift Console's integrated *Observe -> Traces* and *Observe -> Logs* views powered by UIPlugins.
